Task 1. ret2text

实验步骤
GDB

  1. 通过讲解其实已经知道ret2text的源码大概是什么样子了，但还是用IDA看一下

    可以看到在main函数中，只有一个vul()函数

    vul()函数中，这是一个典型的栈溢出漏洞，定义的buf数组只有32个字节，但是实际read()函数会写入64字节，这64字节如果都写满，能够覆盖：

    • 32字节的buf
    • 8字节的saved rbp（通常称为deadbeef）
    • 最重要的是，能够覆盖8字节的函数返回地址

    所以，很容易就能想到，为了能够利用这个栈溢出漏洞，使程序执行backdoor()函数。

    可以在程序执行到vul()函数即将ret的时候，$rsp指向的地址即为函数的返回地址，修改此时rsp指向的地址为backdoor()函数，下一步就能进入backdoor()函数

    理论上来说，只要有backdoor()函数，通过gdb动态调试，可以使任意函数跳转至backdoor()函数，但大多数程序肯定是没有backdoor的

  2. 探究一下backdoor()函数

    execve()函数

    即用 /bin/sh 替换当前进程，拿到shell

  3. 现在来使用GDB进行动态调试，拿到shell

    第一步要做的当然是在vul()函数上打断点b vul

  4. 输入r了之后输入s，让程序单步执行并且能够进入函数，现在应该要执行到read()的位置了

    现在$rsp还是指向vul()函数中

    再s两次，随便输入点什么，执行完read

  5. 输入之后，能够看到刚刚输入的值，以及在汇编语言中，出现了要ret的代码了

    需要的就是执行到ret这个地方，并且将ret指向的值变成backdoor()函数的地址

  6. 输入两次si执行到ret

    可以看到下一步就会返回$rsp指向的函数返回地址

  7. 用set命令，改变$rsp指向的地址，为backdoor()函数

    使用conte stack确实能看出来$rsp现在指向了backdoor函数

  8. si了之后，果然进入了backdoor()函数

    直接c，果然拿到shell了

pwn_tool

  1. 直接观察脚本

    需要在终端tmux中运行，并且会在tmux中新开一个终端，用于进行gdb调试

    以debug的形式输出日志

    要分析的是ret2text这个文件

    自动获取该 ELF 运行时使用的 libc

    main()函数中，首先执行程序，然后根据read()函数中可以填充的字节，构造溢出 payload

    • 输入32个字节的’a’作为buf数组的填充 -> payload = b"a" * 32
    • 填充8字节的saved RBP -> payload += p64(0xdeadbeefdeadbeef)
    • 覆盖返回地址(saved RIP)，这一句pwn_tool能够直接指向文件中的backdoor函数的地址 -> payload += p64(context.binary.sym[‘backdoor’])

    io.sendline(payload)是指交互模式

    新开了一个tmux的终端，并且左边是已经执行完拿到shell的状态，右边是payload还没填充的状态

  3. 直接finish，让val函数执行完，查看寄存器的情况

    能够发现寄存器中，确实存的就是32个字节的a，8个字节的deadbeef，接下来就指向了backdoor()函数，说明read()的栈溢出漏洞利用成功了，该函数的返回地址已经成功覆盖成为了backdoor()函数

Task 2. ret2shellcode

实验步骤

  1. 通过checksec语句可以发现栈的不可执行没有开

    如果NX保护没有开，攻击者很容易可以通过在栈上写入一段shellcode拿到shell，非常危险。

  2. 查看源码

    和ret2text.c的溢出原理差不多，只定义了buf数组0x20个字节，但是read()函数中可以写入0x100个字节，能够通过在栈上写入一段shellcode拿到shell

    并且为了得知栈的地址，还会打印buf的地址

  3. 查看编写的exp_ret2shellcode.py文件

    前面的都和ret2text差不多，通过pwn_tool工具，对ret2shellcode的ELF文件进行处理

    然后，因为源码中打印的buf的地址，所以程序一直读，直到完整得到输出"I will give you stack:"

    然后将栈的地址存在stack_addr变量中，并按16进制解析成整数

    如果成功会输出一条success的日志

    shellcraft.sh()是pwn_tool生成的用于执行 /bin/sh 的汇编代码模板

    接下来就是构造栈溢出payload

    • b"a" * 0x20

      输入0x20个字节的’a’作为buf数组的填充

    • +p64(0xdeadbeefdeadbeef)

      填充8字节的saved RBP

    • +p64(stack_addr + 0x30)

      再往后 8 字节，这里就是函数返回地址(saved RIP)的位置，也就是需要将shellcode的地址写入这8个字节，而shellcode的地址可以就放在这个返回地址之后

      0x30 =48=0x20+8+8，也就是说buf[]数组32个字节+saved RBP 8个字节+saved RIP 8个字节=48=0x30个字节，在这个字节的下一个字节就可以存放shellcode了

    • +shellcode

      payload 的末尾直接跟上生成的shellcode机器码

  4. 运行之后

    可以看到构造的shellcode机器码

    payload执行完之后看一下栈，函数的返回地址指向了下一个地址，也就是shellcode现在的位置

    理论上来说，shellcode可以在任意位置，只要知道地址即可，但是为了方便，一般都放在rip之后

结果展示

返回交互界面，可以看到已经拿到shell了

ret2shellcode完成！

Lab B. Practice

实验步骤

  1. 因为没有源码，所以先用IDA还原出大概的源码

    通过源码观察，至少可以知道，这是有两个栈溢出的漏洞，也就是说，通过name进行一次溢出，再通过buf进行第二次溢出，最终拿到shell

    在ret2libc中，直接打印了libc的地址，所以只需要一次溢出就可以直接拿到shell，但是在这道题里，不知道libc的地址

  2. 尝试用ret2libc的方式获得libc的地址，果然是不行的

    没有任何输出

  3. 所以应该是要通过第一次溢出得到libc的地址；

    再通过第二次溢出实现execve()，最终拿到shell

    但是如果第一次溢出就覆盖了函数的返回地址，就无法进行第二次溢出了

    并且也不能简单地返回main函数，因为每次main函数都会执行

    栈帧会重复搭建，saved RBP再次覆盖，RSP多次下移，read(buf, 0x100) 会写到完全错位的地方，并且第二次溢出会使程序栈segmentation fault

    所以通过搜索，我发现了栈迁移(Stack Pivoting)，即把RSP修改到可控区域。

    可以是.bss段，这是一个内存中不会被破坏的安全区域，如果能够通过第一次栈溢出，使得第二次read()函数的payload写入.bss段，就使第二次溢出能够不使程序栈segmentation fault

  4. 现在构造出两次溢出的ROP

    第一次溢出要实现：找到libc的基地址+栈迁移到.bss段+回到main()函数

    第二次溢出要实现：在.bss段写入execve("/bin/sh", 0, 0)

Step 1. 找到libc的基地址

  1. Hint1中给出了printf(&GOT['printf'])，可以通过这个找到libc的基地址

    通过调用printf函数，并将printf在 GOT 表中的地址作为参数传递，然后printf会读取GOT表里存储的libc库加载到内存后，printf函数的具体位置，并把其打印出来

    后面通过运行程序的基地址=某个函数func的真实地址-函数func的偏移地址就能得到程序的基地址
