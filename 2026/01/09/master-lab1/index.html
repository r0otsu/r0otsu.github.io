    前10个字符需要：

    • 以’0’开头。
    • 以’9’结尾。
    • 包含两个’x’
    • 两个’x’的索引差必须是 4 (例如位置 4 和 8)。
    • 第一个’x’之前的字符串必须包含"key"

    10个字符之后需要：

    • factorial(count(‘1’)) == count(‘1’)，即字符’1’的数量必须是1个或 2 个

  5. 这里直接用Gemini生成Encoder.smali的Java代码

    核心逻辑：

    • Salt (盐): getSalt() 方法生成一个长度为 16 的随机由 “0” 和 “1” 组成的字符串
    • MD5: 将你输入的学号和 Salt 拼接，然后计算 MD5 值（得到 32 位的 Hex 字符串）
    • 混淆 (Encoding): 这是最关键的一步，生成 48 位的字符串
      • 遍历 MD5 字符串和 Salt 字符串
      • 第 1 位放 MD5 的字符
      • 第 2 位放 Salt 的字符
      • 第 3 位放 MD5 的下一个字符
      • 如此循环，把 Salt 像三明治一样夹在 MD5 中间

结果展示

运行之后的完整结果：

使用Soot对LoginScreen.dologin()方法的Jimple代码进行静态分析，发现password的数据流向如下：

  1. Source (输入源)：首先访问Password_Text(EditText)，通过getText()获取用户输入

  2. Flow (数据传递)：获取到的密码字符串被赋值给了类成员变量 password_text 进行暂存

    在发起登录请求前，该变量被再次读取

  3. Sink (数据终点)：分析发现关键函数调用RestClient.doLogin()

    Soot 分析证实，存储密码的变量(password_text)被作为参数直接传递给了RestClient.doLogin()；RestClient负责处理 HTTP 请求，因此确认密码以明文形式流向了网络模块，非常有风险

流向路径为：UI Input -> 成员变量缓存 -> RestClient.doLogin() -> Network
